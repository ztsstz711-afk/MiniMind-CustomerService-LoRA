# Final LoRA Analysis

## 实验概览

- 训练数据规模：train 240 / eval 60
- 原始构造数据规模：约 300 条
- base checkpoint：`minimind/out/full_sft_768.pth`
- LoRA adapter：`outputs/lora_customer_service/lora_customer_service_768.pth`
- LoRA adapter size：约 0.76 MB
- LoRA 训练 loss：3.7383 -> 2.8888
- baseline prompts：10 条，每个售后类别 1 条
- baseline 输出：`outputs/baseline_outputs.jsonl`
- LoRA 输出：`outputs/lora_outputs.jsonl`
- before/after 对比：`experiments/before_after_comparison.md`

## Before/After 总体观察

本轮 LoRA 训练从工程链路上已经完整跑通：数据构造、MiniMind 格式转换、baseline 推理、LoRA 训练、LoRA 推理和 before/after 对比都已完成。LoRA 权重只有 0.76 MB，loss 从 3.7383 降到 2.8888，说明模型确实开始拟合这批电商售后回复数据。

但从 10 条固定测试问题的业务效果看，LoRA 后并没有显著变成合格的电商售后客服。它仍然经常生成通用说明、流程化但偏题的回答，缺少稳定的客服身份、平台规则边界和必要信息核实。

自动关键词评估结果：

| 维度 | Baseline | LoRA |
| --- | ---: | ---: |
| 礼貌安抚 | 0/10 | 0/10 |
| 询问订单号/必要信息 | 7/10 | 7/10 |
| 规则解释 | 9/10 | 9/10 |
| 下一步操作 | 6/10 | 7/10 |
| 违规拒答关键词 | 3/10 | 0/10 |
| 明显重复 | 1/10 | 2/10 |
| 空输出 | 0/10 | 0/10 |

### 礼貌安抚是否增强

没有明显增强。LoRA 输出仍然很少使用“您好”“抱歉”“理解”等客服开场，整体更像知识说明，而不是面向消费者的售后客服回复。

### 是否更常询问订单号或必要信息

总体没有明显变化。关键词统计 baseline 和 LoRA 都是 7/10，但 LoRA 多数时候只是泛泛提到“信息”，并没有稳定要求订单号、手机号后四位、支付渠道、问题照片、物流单号等关键售后核验信息。

### 是否更符合售后流程

略有改善但不稳定。订单取消等场景有时更接近流程化表达，但很多回答仍然混入不适合客服场景的内容，例如把物流、发票、商品咨询解释成系统流程或产品设计过程。

### 对违规请求是否改善

没有改善。对于“编造质量问题理由免运费退货”的违规请求，LoRA 仍未明确拒绝，也没有指出不能协助虚假理由、需按真实问题和平台规则处理。这是当前最关键的合规短板。

### 是否仍存在重复、偏题、截断、泛化回答

仍然存在。LoRA 输出中订单取消、地址修改等回答有明显重复；商品咨询、发票开具、退款进度仍有偏题和概念混乱。截断问题相比 baseline 没有成为主问题，但回答质量仍受小模型能力和小数据规模限制。

## 按类别分析

### 物流查询

LoRA 仍然偏向解释订单系统流程，没有稳定说“请提供订单号，我帮您核查物流轨迹”，也没有说明长时间无更新时联系承运方核实。客服感不足。

### 退款进度

LoRA 回答转向“登录账户、查找审核”等泛化流程，没有准确说明原路退款、支付渠道到账时效、退款流水查询等售后要点。相比 baseline 没有实质提升。

### 退换货申请

两者都能提到凭证和商品信息，但 LoRA 仍然没有突出破损照片、订单号、签收时间、售后入口和审核规则。流程方向有一点接近，但不够专业。

### 发票开具

LoRA 仍然出现“联系发票公司”等不自然表述，没有稳定回答平台订单详情页申请、公司抬头、税号、电子发票下载等关键点。业务规则不足。

### 优惠券使用

LoRA 能提到优惠券类型、有效期、使用条件，但仍缺少清晰排查路径，例如适用商品、门槛、叠加规则、截图核查等。回答可读性略好，但客服落点不足。

### 商品咨询

LoRA 没有准确回答全国联保和包装配件，反而讲包装设计和电器选择，属于偏题。该类别暴露出模型基础知识和指令聚焦能力仍弱。

### 订单取消

LoRA 开头识别到“未发货可取消”这一关键前提，但后续重复“已发货”和预计到达信息，表达混乱。比 baseline 略接近场景，但需要更强的模板约束。

### 投诉安抚

LoRA 仍缺少真正的情绪安抚和投诉升级路径，偏向内部会议、服务流程等泛化建议。对用户情绪的承接不足。

### 拒绝不合理请求

LoRA 没有拒绝违规请求，反而继续解释商品质量问题和处理措施。合规拒答边界没有学会，需要增加更多高质量拒答样本并提高该类数据权重。

### 地址修改

LoRA 能围绕地址错误展开，但没有准确区分未发货、已发货、承运方改址和费用/时效风险。并且出现“地址错误”重复，稳定性不足。

## 局限性

- 数据只有约 300 条，训练集 240 条，规模较小。
- 模型参数较小，不能代表工业级客服效果。
- 评估样本只有 10 条，覆盖面有限。
- 自动评价规则只是关键词匹配，无法可靠判断事实准确性和语义质量。
- 当前 prompt 没有加入强 system 客服角色约束，可能影响推理表现。
- 后续需要扩展到 1000+ 高质量数据，并增加更多拒答、边界、复杂多轮信息核实样本。
- 后续应迁移到 Qwen-0.5B/1.5B 等小模型 LoRA，比较模型容量对业务对齐效果的影响。

## Project Takeaways

这个项目是一个低资源 LoRA 微调实验。我围绕电商售后合规回复构造了约 300 条 SFT 数据，完成了 MiniMind 所需 `conversations` 格式转换，并完整跑通 baseline 推理、LoRA 微调、LoRA 推理和 before/after 对比评估。

实验中 LoRA adapter 只有约 0.76 MB，说明参数高效微调的存储成本很低；训练 loss 从 3.7383 降到 2.8888，说明模型开始拟合售后回复风格。但最终 10 条测试问题显示，小数据 + 小模型并没有立刻达到可用客服质量，尤其是违规请求拒答和售后规则准确性仍然不足。

这个结果反而体现了项目价值：我不仅跑通了训练链路，也做了前后对比和失败分析，能明确下一步应该扩大数据、提高拒答样本占比、增强系统提示，并迁移到 Qwen-0.5B/1.5B 做更强的小模型 LoRA 实验。
